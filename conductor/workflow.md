# Workflow: econ-newsfeed

## TDD Policy

**Strict** — Tests are required before implementation.

- Write failing tests first (red)
- Implement minimal code to pass (green)
- Refactor while tests remain green (refactor)
- No implementation code is considered complete without corresponding tests

## Commit Strategy

**Conventional Commits** — All commits must follow the conventional commit format:

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

**Types:**
- `feat:` — New feature
- `fix:` — Bug fix
- `refactor:` — Code restructuring without behavior change
- `test:` — Adding or updating tests
- `docs:` — Documentation changes
- `chore:` — Maintenance, dependencies, config
- `ci:` — CI/CD changes

**Examples:**
- `feat(api): add GET /api/publications endpoint with pagination`
- `fix(scraper): return bool from fetch_and_save_if_changed`
- `test(api): add integration tests for publication endpoints`

## Code Review

**Required for non-trivial changes.** Trivial changes (typo fixes, config tweaks) can be self-reviewed.

## Verification Checkpoints

**At track completion only.** Manual verification is required when a track is marked complete, not at intermediate phases.

## Task Lifecycle

```
pending → in_progress → completed
                ↓
            blocked (create follow-up task)
```

1. Tasks start as `pending`
2. Move to `in_progress` when work begins (one at a time)
3. Mark `completed` only when fully done and tests pass
4. If blocked, keep as `in_progress` and create a new task for the blocker

## Branch Strategy

- Feature branches per track: `track/{trackId}`
- Merge to `main` after track completion and review
