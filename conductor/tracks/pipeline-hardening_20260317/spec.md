## Summary

Schema fixes, pipeline hardening, and automated scraping infrastructure. Merges DESIGN.md Phases 1, 4, and 5 — widens URL columns, adds unique constraints and indexes, fixes pipeline bugs (return values, duplicate parsing, hardcoded model), adds Pydantic validation for LLM output, creates the scheduler with diff-based extraction, and adds SSRF protection and rate limiting.

## Context

econ-newsfeed scrapes economists' personal websites and uses LLM-based extraction to identify publications. The existing pipeline (html_fetcher.py, publication.py, researcher.py, database.py) has known bugs and missing safety controls that must be resolved before the scheduler and API can be layered on top.

## User Story

As a system operator, I want a reliable and safe scraping pipeline so that new publications are extracted correctly without duplicates, crashes, or security vulnerabilities.

## Acceptance Criteria

- [ ] Schema matches DESIGN.md Section 3.1 (widened URLs, unique constraints, indexes, scrape_log table)

- [ ] Duplicate publications are rejected (normalized title + URL check + DB unique constraint)
- [ ] Scheduler runs scraping on `SCRAPE_INTERVAL_HOURS` interval with threading lock preventing concurrent runs
- [ ] Diff-based extraction sends only changed lines to the LLM
- [ ] SSRF protection rejects private IPs, non-HTTP(S) schemes, and metadata endpoints
- [ ] Per-domain rate limiting and exponential backoff on retries are implemented
- [ ] `.env.example` documents all env vars; app fails fast if required vars are missing
