**Type:** Feature

## Summary

FastAPI REST API serving publication and researcher data to the frontend, with scrape trigger/status endpoints, CORS, security headers, and standard error responses. Implements DESIGN.md Phase 2 and Section 4.

## Context

The econ-newsfeed system needs a REST API layer between the Next.js frontend and the MySQL database. DESIGN.md Section 4 fully specifies the endpoints, response shapes, error envelope, CORS policy, and security headers. The API is the second implementation phase — it depends on the pipeline fixes and data layer from Track 1 (pipeline-hardening_20260317).

## User Story

As a frontend application, I want a REST API that serves paginated publications and researcher data so that the newsfeed UI can display up-to-date content.

## Acceptance Criteria

- [ ] `GET /api/publications` returns paginated results (`page`/`per_page`) sorted by `timestamp DESC`, with optional `year` and `researcher_id` filters
- [ ] `GET /api/publications/{id}` returns a single publication with authors, or 404
- [ ] `GET /api/researchers` returns all researchers with URLs and publication counts
- [ ] `GET /api/researchers/{id}` returns researcher profile + publications, or 404
- [ ] `POST /api/scrape` requires `X-API-Key` header, returns 401/409/201 as specified
- [ ] `GET /api/scrape/status` returns last scrape info and next scheduled time
- [ ] All errors use standard envelope: `{"error": {"code": "...", "message": "..."}}`
- [ ] CORS allows only `FRONTEND_URL` origin
- [ ] Security headers (`X-Content-Type-Options`, `X-Frame-Options`, CSP, HSTS) on all responses
- [ ] OpenAPI docs available at `/docs`

## Dependencies

- **Track 1 (pipeline-hardening_20260317):** Provides fixed schema, data layer functions (`database.py`, `researcher.py`, `publication.py`), and `scheduler.py` with `run_scrape_job()` and scrape lock.

## Out of Scope

- Authentication beyond API key for scrape endpoints
- Rate limiting on read endpoints
- WebSocket/SSE for real-time updates
- GraphQL
- Admin endpoints for managing researchers

- Uses FastAPI lifespan handler to start scheduler and run `Database.create_tables()` on startup
- Pydantic response models for all endpoints (auto-generates OpenAPI docs)
- Scrape endpoint runs job in background thread via `scheduler.py`'s `run_scrape_job()`
- Imports from `database.py`, `researcher.py`, `publication.py`, `scheduler.py` — all fixed by Track 1
